# Library

Консольное приложение для управления небольшой библиотекой, разработанное с соблюдением принципов чистой архитектуры, SOLID и объектно-ориентированного проектирования.

---

## Доступные команды

| Команда  | Описание                                      | Использование                    |
|----------|-----------------------------------------------|----------------------------------|
| `ADD`    | Добавить новую книгу                          | `ADD Название;Автор;Год`         |
| `REMOVE` | Удалить книгу по UUID                         | `REMOVE <uuid>`                  |
| `LIST`   | Показать книги (с опциональной сортировкой)   | `LIST [title \| author \| year]` |
| `FIND`   | Поиск по названию или автору                  | `FIND запрос`                    |
| `STATS`  | Показать статистику библиотеки                | `STATS`                          |
| `EXIT`   | Завершить работу программы                    | `EXIT`                           |

##  Архитектура и принятые решения

Я построила приложение на строгой слоистой архитектуре (Runner → Handler → Command → Controller → Service → Repository → Writer), чтобы чётко разделить зоны ответственности и не смешивать бизнес-логику с консольным вводом-выводом. Каждый слой занимается только своей задачей, а именно: Runner управляет вводом и запускает обработку, обработчики распознают команды и проверяют синтаксис, сервисы валидируют бизнес-правила, контроллер оркеструет выполнение и перехватывает ошибки, а Writer полностью берёт на себя формат вывода. Зависимости передаются через конструкторы, а взаимодействие идёт только через интерфейсы. Благодаря этому компоненты остаются изолированными, и их легко покрывать юнит-тестами с моками, не запуская реальную консоль и не трогая соседние слои.  

Для маршрутизации я выбрала паттерн «Цепочка обязанностей», а внутри команды LIST реализовала вложенную цепочку для сортировок. Это необходимо, чтобы соблюдать принцип открытости/закрытости. Для того, чтобы добавить новый вид сортировки (да и в целом любой команды), достаточно создать один обработчик и подключить его к цепочке, не меняя существующую логику маршрутизации. Исключения бизнес-уровня пробрасываются до контроллера, где отлавливаются и преобразуются в понятные пользователю сообщения, не прерывая работу приложения. В итоге система получилась гибкой, расширяемой и лёгкой для отладки и тестирования.

## Инструкция по запуску

1. Убедитесь, что на компьютере установлены **Java 22+** и **Maven 3.8+**.
2. Клонируйте репозиторий и перейдите в корень проекта:
   ```bash
   git clone <your-repo-url>
   cd Library
   ```
3. Соберите и запустите приложение:
   ```bash
   mvn clean compile exec:java
   ```
   *(Альтернативно: откройте проект в IntelliJ IDEA и запустите `Main.java` напрямую.)*

---

## Тестирование

Проект покрыт модульными тестами на **JUnit 5** с использованием **Mockito**.
- `LibraryControllerImplTest` - проверка корректной координации слоёв и обработки ошибок (сервис и `Writer` замокированы).
- `HandlerChainTest` - проверка маршрутизации команд, валидации формата ввода, работы вложенной цепочки и делегирования вызовов.

Запуск всех тестов:
```bash
mvn test
```
